<이기적CBT 오답 & 헷갈리는 개념 정리>

1. 파생속성 - 다른 속성에 영향을 받아 발생하는 속성으로서 보통 계산된 값들이 이에 해당, 다른 속성에 영향을 받기 때문에 프로세스 설계 시 데이터 정합성을 유지하기 위해 유의해야 할 점이 많음. 가급적 속성을 적게 정의하는 것이 좋음

2. 엔터티는 반드시 하나 이상의 속성을 가져야 의미가 있음. 속성이 없으면 데이터가 존재하지 않는 것이므로 엔티터로서 성립하지 않음

3. 트랜젝션 제어어 - COMMIT / DCL - GRANT / 

4. ERD 작성순서
  1) 엔터티를 그림 2) 엔터티를 적절하게 배치 3) 엔터티간에 관계를 설정 4) 관계명을 기술 5) 관계의 참여도를 기술 6) 관계의 필수 여부를 기술

5. 그룹 함수를 사용하는 경우 NULL 값은 연산에서 제외. 그래서 NVL 함수를 사용하는 것은 합계오류예방과 전혀 관계 X

6. 개념적 데이터 모델링 단계의 주요 산출물 - ERD : 실세계의 개체, 관계등을 도식화

7. 조건에서 NULL = NULL 은 공집합 나옴(데이터 X) => 문법적 오류는 아님 그렇지만 IS NULL로 비교하는게 맞음!

8. 정규화의 이점 : 데이터중복 최소화, 이상 현상 제거, 무결성을 보장/ 지나친 정규화는 JOIN을 증가시켜 성능을 저하시킬 수 있음

9. 참조 무결성 - 외래키는 기본키 없는 자식이어서는 안된다는 원칙
		    - 자식 테이블의 부서번호는 반드시 부모테이블의 부서번호에 실제로 존재하는 값이거나 아예 비어 있어야 함,
    * 위배 사례 : 사원테이블에 데이터를 입력할 때, 부서테이블의 부서번호에 없는 번호를 입력하여 추가하면 참조 무결성을 위배했다고 함.
   ** 자식의 데이터를 지울 땐 상관 없지만, 부모를 지울 땐 자식이 고아가 될 수 있어 문제가 생기는 것을 참조무결성이라고 봄.(참조할게 사라져서)

10. 개체 무결성 - 부모테이블에 이미 있는 부서번호를 입력하여 부서번호가 겹치는 상황
		      - 한 테이블의 기본키는 중복될 수 없으며, NULL 도 안됨

11. 도메인 무결성 - 특정 속성의 값은 정의도니 데이터 타입/ 범위 내여야 함.

   ** 자식이 사고치면 참조 무결성, 부모내에서 번호가 겹치면 개체 무결성

12. Sort Merge 방식의 조인이 Nested Loop 방식 조인보다 효율적으로 판단되는 것 
	: 기본키와 외래키 관계에서 외래키에 인덱스가 없을 때 
	- Sort Merge 방식의 조인은 두 개의 테이블을 Sort 한 후에 Merge를 한다. Merge가 완료되면 한 번의  Full Scan으로 데이터를 검색함. 기본키와 외래키 관계에서 외래키에 인덱스가 없을 때 옵티마이저가 Sort Merge 방식의 조인으로 동작하게 함.

23. Hash Join 특징
 - 대용량 데이터를 조인할 때 성능이 좋아서 자주 쓰이는 방식
 - 대용량 데이터 조인 시 속도가 매우 빠름
 - 정렬되지 않은 데이터도 상관없음
 - 조인 컬럼의 인덱스가 존재하지 않을 경우에도 사용가능
 - 해시 함수를 이용하여 조인을 수행하기 때문에 '='로 수행하는 조인인 동등 조건에서만 사용가능
 - 해시 함수가 적용될 때 동일한 값은 항상 같은 값으로 해시됨을 보장
 - 작업을 수행하기 위해서 해시 테이블을 메모리에 생성
 - 해시 테이블을 저장할 때 메모리에 적재할 수 있는 영역의 크기보다 커지면 임시 영역(디스크)에 저장
 - 해시 조인을 수행할 때는 결과 행의 수가 적은 테이블을 선행 테이블로 사용하는 것이 좋음
 - 선행 테이블을 Build Input 이라고 하며 후행 테이블은 Prove Input 이라 함

WITH 절(CTE, Common Table Expression)은 SELECT 문 시작 전에 미리 정의된 가상의 테이블(임시 결과셋)을 생성하는 문법이다. 이는 마치 FROM 절에 인라인 뷰를 사용하는 것과 유사한 개념이며, 복잡한 쿼리를 구조화하고 가독성을 높이는 데 유리하다. 또한, 같은 CTE를 여러 번 재사용할 수 있는 장점이 있다. 만약 한 번만 사용할 용도라면 인라인 뷰로 작성해도 무방하다. 이 문제에서 CTE REGION_TOTAL은 TB_ORDER 테이블을 REGION 별로 그룹화한 뒤, 각 지역의 주문 금액 합계를 계산한다. 이후 메인 쿼리에서는 TOTAL > 1500 조건을 만족하는 지역만 필터링하여 COUNT(*)를 수행한다. 조건을 만족하는 지역은 EAST 하나뿐이므로, 최종 결과는 1건이다.​​​

34. 
- Access Subquery : 서브쿼리가 메인쿼리의 제공자 역할을 하고, 메인쿼리의 값이 서브쿼리에 주입되지 않음.
- Associative Subquery : 상관 서브쿼리, 메인쿼리의 한 행이 읽힐 때마다 서브쿼리가 그 값을 받아와서 실행 ex) 반장(메인)이 학생이름을 한명씩 부를 때마다 서기(서브쿼리)가 그 학생의 성적을 생기부에서 찾아오는 방식
- Early Filter형 Subquery : 더 빨리, 조인이나 무거운 연산을 하기 전에 서브쿼리를 먼저 실행해서 불필요한 메인 쿼리의 행들을 미리 제거, 처리해야 할 데이터 양 자체가 줄어드니 뒤에 이어지는 연산 속도가 비약적으로 빨라짐.
- Filter형 Subquery : 필터형 서브쿼리, 메인 쿼리에서 추출된 데이터에 대해 서브쿼리를 마치 WHERE절의 필터처럼 사용하는 것, 메인쿼리의 결과 행마다 서브쿼리를 실행, 매번 실행하면 느리니까 서브쿼리 캐싱을 활용하여 한번 계산한 값은 메모리에 저장했다가 똑같은 값이 들어오면 재사용

40. 윈도우 함수
 - GROUP BY는 특정 컬럼 값을 기준으로 행을 그룹화하여, 집계된 결과만 출력하므로 출력 행수가 줄어드는데, 윈도우 함수는 행수를 줄이지 않고, 각 행에 대해 집계 결과나 순위 등의 값을 추가로 계산함.
 - PARTITION BY 컬럼; 을 하면 각 컬럼의 값에 대해 윈도우 행 범위를 넘어갈 수 없음
 - WINDOWING절을 사용하려면 ORDER BY가 있어야 함
 - PARTITION BY절이 없으면 전체 행을 대상으로 진행

<아이리포 CBT 오답 & 헷갈리는 부분 정리>
1. 인덱스
 - INSERT, UPDATE, DELETE 등과 같은 DML 작업은 테이브로가 인덱스를 함께 변경해야 하기 때문에 오히려 속도가 느려질 수 있음.
 - 인덱스 사용 목적은 검색기능 최적화
 - 인덱스 데이터는 인덱스를 구성하는 컬럼의 값으로 정렬을 수행
 - 인덱스 중 B-Tree 인덱스는 Equal 조건 외에도 BETWEEN, <, >등 범위 검색도 가능

2. 단일컬럼 IN vs 다중컬럼 IN
 - 단일컬럼의 IN : OR의 느낌 
    ex) WHERE COL1 IN(10000, 20000) => COL1 = 10000 OR COL1  = 20000

 - 다중컬럼의 IN : 세트의 느낌, 행 단위의 완전한 일치
    ex) (COL1, COL2) IN ((10000, 'ABC')) => COL1 10000 AND COL2 = 'ABC' 인 세트를 찾아라! 

3. 실행계획 : SQL을 실행하기 위한 절차와 방법을 말함

4. 테이블 생성 후 복제(CTAS)에서 제약 조건 적용 : PK는 복제되지 X
 -> CTAS : CREATE TABLE AS SELECT를 말하며 NOT NULL 제약조건만 복사, CHECK, DEFAULT 모두 적용 X

- 로우체이닝 : 한 테이블에 너무 많은 컬럼이 존재할 경우 한 행의 크기가 너무 커서 디스크의 데이터 블록 하나에 저장되지 못하고 두 개 이상의 블록에 걸쳐서 저장이 되는 것
- 파티셔닝 : 하나의 테이블에 저장된 데이터가 너무 많아서 전체를 검색하는데 성능이 심각하게 저하되는 경우 논리적으로 하나인 테이블을 동일한 스키마를 가진 여러 테이블로 나눠 저장하여 조회성능을 향상시키는 반정규화 기법
 - 클러스터링 : 여러개의 데이터베이스 서버를 하나로 묶어서 사용하는 기법, 이때 여러 개의 서버는 하나의 저장소를 공유하며 작업을 나눠 처리하는 방법에 따라 Active - Active, Active-Standby 방식으로 나뉨
 - 샤딩 : 수평 분할의 일종으로 동일한 스키마를 가진 여러 대의 데이터베이스에 데이터를 분산 저장하는 방법, 파티셔닝은 나눠진 테이블을 동일한 디비에 저장하는데 비하여 샤딩은 나눠진 테이블을 네트워크에 분산 저장한다는 것이 다름.

 - 제 1정규화의 경우 모든 속성이 원자성을 확보하여 주식별자에 대해 1:1관계를 가지도록 하는 것
 - 제 2정규화는 일반 속성 중 주식별자의 일부에 대해 함수종속성을 갖는 부분 함수종속성을 제거하는 것
 - BCNF : 보이스-코드 정규화, 후보키가 아닌 속성이 결정자인 함수종속성이 존재하는 경우에 수행하는 저규화, 일반적으로 일반 속성 중 하나가 일부 주식별자의 결정자가 되는 경우에 해당하므로 주식별자와 관련이 있음.
