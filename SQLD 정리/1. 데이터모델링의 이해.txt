<데이터 모델의 이해>
1. 모델링
 1) 모델링 개념 
 - 현실 세계의 데이터를 약속된 표기법으로 표현, 업무 프로세서의 데이터를 분석하여 추상화
   -> 업부의 흐름을 가시화하고 명세화하여 설계, 개발 또는 시스템 관리에 사용하기 위한 것
 - 추상화를  기본으로 함
 2) 모델링 특징
 - 추상화(Abstraction) : 주요 특징을 추출하여 일정한 형식으로 표현
 - 단순화(Simplirfication) : 약속도니 표기법을 사용하여 누구나 이해하기 쉽도록 함. ex) ERD
 - 명확화(Clarity) : 보는 사람에 따라 모호함이 없게 기술하는 것

 * 모델링 시 유의사항
 - 중복 최소화 : DB의 여러 곳에 같은 정보를 중복하여 저장하지 X -> 데이터 불일치 현상 일어날 수 있기 때문임
 - 비유연성 최소화 : 데이터의 정의와 데이터의 사용 프로세스를 분리하여 데이터 또는 프로세스의 변화가 응요 프로그램과 데이터베이스에 미치는 영향을 최소화 함. -> 데이터 처리, 정의 분리시키는게 좋음
 - 비일관성 최소화 : 데이터 간의 상호 연관관계를 명확하게 정의하여 데이터가 일관성 있게 유지되어야 함.

 3) 모델링의 세가지 관점
  - 데이터관점(대상,  What) : 업무를 구성하는 데이터에 집중하여 어떤 데이터들이 서로 관계를 맺고 사용되는지를 모델링, 정적분석, 구조분석 등을 기반
  - 프로세스 관점(처리방법, How) : 업무의 흐름에 집중하여 업무가 실제로 처리하는 일이 어떻게 처리되는지를 모델링, 동적분석, 도메인 분석 등을 기반
  - 데이터와 프로세스의 상관 관점(대상과 처리방법의 상관관계, Interaction) : 업무를 구성하는 데이터와 프로세스가 서로 어떻게 관계를 맺고 영향을 주고 받는지를 모델링, CRUD(Create, Read, Update, Delete)분석을 기반

 4) 모델링의 세가지 단계 : 개 -> 논 -> 물
  (1) 개념적 데이터 모델링 : 전체, 포괄, 전사적
    -> 가장 높은 추상화 레벨
    -> 포괄적인 수준에서 모델링을 수행
    -> EA(Enterprise Architecture)를 수립할 때 많이 이용되며, 이 단계에서 엔터티와 속성을 도출

  (2) 논리적 데이터 모델링 : 관계, 정규화, 재사용
    -> 데이터 모델에 대한 키, 속성, 관계 등을 표현
    -> 서로 다른 DBMS에 적용이 가능한 수준에서의 추상화 레벨을 가지므로 재사용성이 높음
    -> 정규화를 통해 중복 데이터를 최소화
    -> 식별자를 도출하고 관계를 정의

  (3) 물리적 데이터 모델링 : 디스크, 메모리, 성능, 보안, 가용성, 테이블, 인덱스, 함수
    -> 특정 DBMS에 맞추어 구현이 가능한 수준에서 모델링
    -> DBMS의 성능이나 보안, 가용성 등을 고려하여 설계
    -> 가장 낮은 수준에서의 추상화 레벨을 가짐.
    -> 성능향상을 위해 반정규화를 수행하며, 테이블 인덱스, 함수 등을 생성

 5) ANSI-SPARC 에서 정의한 3단계 스키마구조
  - 데이터 독립성을  보장 : 데이터를 사용하는 사용자 영역과 실제로 데이터를 저장하는 디스크나 메모리 영역을 서로 분리하여 상호 간에 데이터 구조나 형식에 있어서 독립적으로 설계되고 운용될 수 있도록 하는 특성을 말함. -> 각 계층별 의존성이 감소하여 전체 시스템의 유지보수성이 향상되고, 시스템변경 작업 시 작업기간 단축, 비용 감소 등의 이점이 있음 

  (1) 외부 스키마 : 사용자 관점,  사용자 or 애플리케이션이 바라보는 데이터베이스 스키마를 정의, 다중 사용자 뷰를 제공
  (2) 개념 스키마 : 통합, 설계자 관점, 모든 사용자가 바라보는 데이터베이스 스키마를 통합하여 나타냄. 통합된 뷰를 제공
  (3) 내부 스키마 : 개발자 관점, 디스크나 메모리 상의 물리적, 실질적 저장구조를 나타냄. 테이블, 칼럼, 인덱스 등을 정의, 물리적 뷰를 제공

 * 외부 -> 개념 : 논리적 데이터 독립성을 보장
   개념 -> 내부 : 물리적 데이터 독립성을 보장

 * 3단계 스키마 구조를 통해 보장되는 데이터 독립성 
   - 의존성 낮음
   - 논리적 데이터 독립성 : 외부 스키마 -> 개념 스키마 간의 독립성. 개념 스키마가 변경되어도 외부스키마는 영향 X
   - 물리적 데이터 독립성 : 개념 -> 내부 스키마 간의 독립성. 내부 스키마가 변경되어도 개념 스키마나 외부 스키마는 영향X

 6) ERD  
  - 데이터 베이스를 구성하는 데이터의 논리적 구성요소를 엔터티로 정의, 엔터티들 간의 고나계를 특별한 표기법으로 나타냄으로써 스키마를  설계하는 방법을 제공
 - 데이터 모델리에 대한 문서화 방법을 제공
 - 시스템분석, 설계의 결과를 가시화, 문서화, 명세화하여 사용자, 설계자, 개발자 간의 의사소통 수단이 됨
 - 엔터티 : 박스로 나타냄, 엔터니 이름은 박스 상단에 표시
 - 속성 : 박스 안에 리스트 형식으로 표현, 주식별자(PK)와 일반 속성은 칸막이로 구분
 - 관계 : 박스를 연결한 선으로 표현, 식별자 관계는 실선, 비식별자 관계는 점선으로 표현
 - 카디널리티 : 까치발이라고 부르는 관계, O는 0, 세로 실선은 1, 세 개의 까치발 모양은 다수(2 이상)을 의미
                   : 특정 테이블 컬럼 내에 존재하는 고유한 값의 갯수
                   : 높은 카디널리티 - 고유값의 수가 많음 ex) ID, 이메일, 주민번호 -> 인덱스 효율 좋음
                   : 낮은 카디널리티 - 고육값의 수가 적음 ex) 성별, 학년, 상태값 -> 인덱스 효율 낮음
                   : 관계형 카디널리티 - ERD에서 엔터티 간의 관계를 나타냄, 1:1, 1:N(1대 다), N:M(다대다)

 - ERD 작성순서
  -> 엔터티 도출 
  -> 도출된 엔터티를 적절하게 배치, 중요한 엔터티는 오니쪽 상단에 배치
  -> 엔터티 간의 관계를 설정
  -> 관계명을 기술(행위관계, 존재 관계를 표현)
  -> 관계의 참여도를 기술(일대일, 일대다, 다대다), 중요하기에 생략 X
  -> 관계의 필수/선택 여부를 기술(NULL 값을 가질 수 있는지 여부)
 
2. 엔터티
 1) 엔터티 개념 
 - 데이터베이스의 구성요소 중 독립적으로 식별 가능한  개체 
   ex) 회원, 대출, 도서 
 - 엔터티는 보다 상세히 자신을 표현하기 위해서 속성이라는 하위요소를 가짐
   ex) 회원명, 주소, 전화번호, 아이디, 비밀번호
 
 * 주식별자를 다른 말로 PK(PRIMARY KEY, 기본키)라고 부름, 엔터티를 ERD로 표현할 때 PK 속성과 일반 속성은 칸막이로 구분
 2) 특징
 - 업무에서 필요로하고 관리하고자 하는 정보
 - 유일한 식별자
 - 인스턴스(디스크에 저장된 데이터)가 2개 이상인 집합을 이루어야 함
 - 하위요소로 반드시 속성을 가져야 함
 - 엔터티는 다른 엔터티와 한 개 이상의 관계를 가져야 함.(통계성 엔터티나 코드성 엔터티의 경우 관계 생략 가능)
 
3) 분류
(1) 발생 시점/ 상속관계에 따른 분류
 - 기본 엔터티 : 자신의 고유한  주식별자를 가지는 독립적으로 생성되는 엔터티 ex) 사원, 부서, 고객, 상품, 자재
 - 중심 엔터티 : 기본 엔터티로부터 주식별자를 상속받아 생성되며 업무의 중심 역할을  하는 엔터티 ex) 급여, 주문
 - 행위 엔터티 : 두 개 이상의 엔터티를 상속받아 생성되는 엔터티, 내용이 자주 변경되거나 데어타 양이 계속 증가 ex) 급여내역, 주문내역
 
(2) 물리적 형태의 존재 여부에 따른 분류
 - 유형 엔터티 : 물리적 형태가 존재 ex) 사원, 상품, 학생
 - 개념 엔터티 : 물리적 형태 없이 개념적으로 정의 ex) 부서, 상품분류, 강의
 - 사건 엔터티 : 행위나 이벤트를 나타내는 엔터티 ex) 주문, 대여, 수강

 (3) 교차엔터티 : M:N(다대다) 관계를 해소하기 위해 만들어진 엔터티

 * 엔터티 명명법
 -> 엔터티 생성 의미대로 이름을 부여
 -> 약어를 사용 X
 -> 대문자 사용, 단수명사 사용
 -> 업무상의 의미를 분명하게 사용
 -> 이름은 구별될 수 있도록 유일
 -> 의미상 중복 X

3. 속성
 1) 개념
 - 엔터티에 대한 구첵적인 정보를 나타냄, 엔터티의 하위요소
 - 업무와 관계되어 필요한 것들로 정의
 - 속성의 개수는 제한 X, 필수적인 것들로 최소화해야 함
 - 속성값들의 집합
 - 최소의 데이터 단위

 2) 속성값
 - 속성이 가질 수 있는 특정 값(하나의 데이터)
 - 하나의 엔터티 인스턴스에서 한 개의 속성값만을 가져야 함
 - 한 개 이상의 속성값을 가질 수 있는 경우라면 1차 정규화를 수행해야 함
 - 단일 정보만 담는 것이 좋음

 3) 엔터티(테이블), 인스턴스(행), 속성(열), 속성값(하나의 데이터)
 - 한 개의 엔터티는 두 개 이상의 인스턴스를 가짐(엔터티는  인스턴스의 집합)
 - 한 개의 엔터티는 두 개 이상의 속성을 가짐(엔터티는 속성의 집합)  
 - 한 개의 속성은 한 개의 속성값만 가짐
 - 인스턴스가 한 개인 경우라면 굳이 엔터티로 정의할 필요 X

 4) 속성의 분류
 (1) 속성의 특성에 따른 분류
  - 기본 속성 : 엔터티가 보낼부터 가지고 있어야 하는 속성
  - 설계 속성 : 설계 시 필요하다고 판단되어 도출된 속성
  - 파생 속성 : 다른 속성으로부터 계산되거나 특정 규칙에 따라 변형되어 만들어진 속성

 (2) 구성방식에 따른 분류 
  - PK(PRIMARY KEY) : 기본키, 해당 엔터티의 인스턴스를 유일하게 식별할 수 있는 속성
  - FK(FOREIGN KEY) : 외래키, 관계를 통해 다른 엔터티의 속성을 가져와 포함시킨 속성
  - 일반 : 키가 아닌 나머지 일반

 5) 도메인
 - 속성이 가질 수 있는 값의 범위(값의 데이터 타입과 크기)
 - 데이터 타입, 크기, 제약사항 등을 묶어 별도의 이름을 붙여 정의, 각각의 속성에 지정
 * 시스템카탈로그 : DB에 저장되어 있는 모든 데이터 객체들에 대한 정의나 명세에 대한 정보가 수록되어 있는 시스템 테이블
 * 카디널리티 : 데이터 집합의 유일한 값의 개수
