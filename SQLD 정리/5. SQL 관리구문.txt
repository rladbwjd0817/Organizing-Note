<SQL 관리구문>

1. DML : 데이터 조작어
 - SELECT : 조회

 - INSERT : 삽입
    [기본 문법]
    -> INSERT INTO 테이블명 (컬럼명1, 컬럼명2, ...) VALUES(데이터1, 데이터2);
    ->1대 1로 대응되며 리스트에 없는 컬럼은 NULL 데이터 들어감
    -> 컬림리스트 생략 시 입력할 데이터는 컬럼의 순서에 맞게 입력해야 그 순서에 맞게 데이터가 들어감

 - UPDATE : 수정
    - WHERE절을 통해 수정하고자 하는 레코드를 특정하며 지정된 컬럼의 데이터 수정
    [기본 문법]
    -> UPDATE 테이블명 SET 컬럼1 = 데이터1, 컬럼2 = 데이터2 ... WHERE 조건식;

 - DELETE : 삭제
   - TRUCATE는 작업취소를 위한 데이터를 생성하지 않아 속도가 빠르고 디스크 공간을 릴리즈하여 재상용이 가능하게 함.
   - DELETE는 로그를 남기며 롤백이 가능함
   [기본 문법]
    -> DELETE FROM 테이블명; =========> 해당 테이블의 데이터 전체 삭제
    -> DELETE FROM 테이블명 WHERE 조건식; ====> 해당 테이블에서 조건에 맞는 데이터만 삭제 

 - MERGE : 갱신
   - 두 테이블을 비교하여 특정 조건에 맞는 레코드에 대해서 UPDATE문을 실행하여 데이터를 갱신하고, 그렇지 않은 레코드는 INSERT문ㅇ르 사용해 신규 추가 
   - 기존 테이블의 변경사항을 백업 테이블에 반영하거나 개발을 위해 변경 또는 추가할 레코드를 운영 환경에 적용할 때 사용 가능
   [기본문법]
    -> MERGE INTO 테이블명 USING 테이블명2 ON 조건식1
    -> WHERE절을 사용해서 특정 조건의 행에 대해 UPDATE를 수행하도록 할 수 있음.
    -> DELETE절은 UPDATE가 된 행에 한아여 조건에 따라 실행되므로 주의
    -> WHEN MATCHED THEN과 WHEN NOT MATCHED THEN 구문은 둘 중에 하나만 사용하는 것도 가능. 이런 경우 사용되지 않는 구문에 대해서는 아무런 동작을 수행하지 X

 * Oracle 기준 COMMIT은 모드에 따라 다르기 때문에 기본모드가 Off 이므로 명시적으로 커밋을 수행해야 하고, 롤백도 가능.
 * ROLLBACK : 명시적으로 커밋을 실행하기 전에는 작업을 되돌리는 것이 가능

2. TCL : 트랜잭션 제어 언어
 - 트랜잭션 : 앞에서 데이터베이스에 데이터를 읽고 쓸 때, 한번에 수행되어야 하는 논리적인 작업 단위

   ex) 은행 타인에게 계좌이체 : 유정이가 희정이에게 10000원을 입금을 했는데 그 순간 정전이 발생되었다. 
        그럼 이때 유정이 통장에서는 10000원이 빠져나가고, 희정이 통장에 10000원이 입금이 안되면 문제가 발생!!!!!
        이러한 문제를 막기 위해서 트랜잭션을 실시 -> 두 가지 기능이 다 완료되면 찐으로 완료되고, 하나라도 완료되지 않으면 롤백하는거(원자성)

 1) 특징 : 원 - 일 - 고 - 영
  (1) 원자성 : 하나의 트랜잭션으로 묶인 연산자들은 All or Nothing의 개념으로 모두 실행되는지 아니면 전혀 실행되지 않아야 함
  (2) 일관성 : 트랜잭션의 결과는 디비의 정합성을 깨지 않는다는 것으로서 트랜잭션 이전에 디비에 오류가 없다면 트랜잭션 이후에도 오류가 없음
  (3) 고립성 : 트랜잭션은 독립적으로 수행되며 다른 트랜잭션이 실행 중간에 간섭하거나 영향을 미치지 X
  (4) 영속성 : 트랜잭션의 결과는 디비에 영구적으로 저장되어 있어야 함.(회원탈퇴해도 데이터는 디비에 남아있음.....wow..)
 
 2) COMMIT : 저장
 - COMMIT 명령어는 INSERT, UPDATE, DELETE와 같은 DML 명령들을 통한 변경사항을 디비에 영구적으로 반영하고 락을 해제하여 트랜잭션을 완료함.
 - 락 : 트랜잭션을 시작하면 일종의 쓰기 잠금을 발생하여 다른 사용자가 디비의 데이터를 수정, 반영하는 것을 막는 것.
 
 3) ROLLBACK : 되돌리기(Ctrl + Z)
 - 트랜잭션에 포함되는 전체 변경사항(이전의 커밋 명령 이후의 변경사항) 또는 지정된 저장점(SAVEPOINT) 이후의 변경사항을 취소하고 원래대로 되돌림
 - 커밋과 마찬가지로 락이 해제
 - 이미 커밋한건 롤백 안됨 
   -> CREATE, ALTER, TRUNCATE와 같은 DDL명령어는 자동 커밋이 되기에 롤백 불가능
   -> INSERT, UPDATE, DELETE같은 DML명령어는 자동 커밋이 되지 않아 롤백 가능

 * SQL Server의 경우 자동커밋 몸드가 되면 DML명령어도 자동 커밋이 되어 롤백 불가능, BEGIN TRANSACTION에 의한 명시적 트랜잭션에 대해서는 롤백이 가능

 4) SAVEPOINT
 - ROLLBACK하기 위한 저장점 지정
 - 저장점 이후에 해당하는 변경사항만 취소됨
  [Oracle 기본 문법] --------------------------------> 오라클에선 특별히 트랜잭션의 시작을 지정하지 X, DML코드가 시작되면 자동으로 트랜잭션이 시작
    -> SAVEPOINT <이름>;
	 ROLLBACK TO <이름>;
  [SQL Server] ------------------------------------> BEGIN TRANSACTION으로 트랜잭션의 시작을 명시적으로 지정 가능
    -> SAVE TRANSACTION <이름>;
	 ROLLBACK TRANSACTION <이름>;
      

3. DDL : 데이터 정의어
 - 스키마를 정의하고 이를 관리하는 명령어
 - ERD로 그려진 데이터 모델링은 논리적 모델링으로서 엔터티와 속성, 관계를 특정 DBMS에 의존적이지 않게 설계하는 것
 - DDL을 이용해 특정 DBMS에 실제로 적용하는것을 물리적 모델링
 - 테이블 생성할 때 지정해야 할 요소
    -> 테이블 이름, 컬럼명, 컬럼 데이터 타입, 컬럼 데이터 크기, 제약조건

 1) CREATE : 테이블 생성
   [기본 문법]
   CREATE TABLE 테이블명( 
	컬럼명1 데이터타입, 제약조건
    );

   - 테이블 생성시 주의할 점 
	: 테이블 이름은 해당 데이버테이스 내에서 고유해야 함(중복 X)
	: 컬럼명도 중복 X
	: 컬럼에 대한 이름, 데이터 타입은 필수 지정
	: DEFAULT, NULL, NOT NULL 등 선택사항, 지정하지 않으면 기본값은 NULL 지정
	: 테이블명, 컬럼명, 제약조건 이름 등 이름을 지정하는 경우 문자만 허용
	: 컬럼의 정의는 괄호안에 기술
	: 각 컬럼은 콤마로 구분
	: 세미콜론으로 끝냄
	: 테이블명을 지정할 때 의미없이 짓지 X
	: 컬럼의 이름은 통일성을 살려서 짓기
 
   - 제약조건(CONSTRAINT)
    (1) PRIMARY KEY(기본키, PK) : NULL 값 가질  수 X, 모든 값이 고유해야 함. 자동으로 UNIQUE INDEX 생성
    (2) FOREIGN KEY(외래키, FK) : PK로부터 가져온 컬럼을 지정, 논리적 모델링에서 정의한 관계를 표현, FK 지정시 참조 대상인 컬럼에 대해서 참조 무결성 제약조건 지정 가능

    * 참조 무결성 제약조건 
    -> DELETE/MODIFY ACTION
	- CASCADE : 부모의 값 삭제 시 자식의 값도 같이 삭제(행 전체가 삭제)
	- SET NULL : 부모의 값 삭제 시 자식의 해당 컬럼은 NULL값(해당 컬럼의 값만 삭제)
	- SET DEFAULT : 부모의 값 삭제 시 자식의 해당 컬럼은 기본 값으로 변경
	- RESTRICT : 자식 테이블에 해당 데이터가 PK로 지정되지 않은 경우에만 부모 테이블에서 삭제 및 수정이 가능
	- NO ACTION : 제약 조건을 위배한 경우 동작 실패 => 아무 액션 취하지 않음

    -> INSERT ACTION 
	- AUTOMATIC : 부모 테이블에 PK가 없는 경우 PK 생성 후 자식 테이블에 값이 입력
	- SET NULL : 부모 테이블에 PK가 없으면 자식 테이블에 NULL 값
	- SET DEFAULT : 부모테이블에 PK가 없으면 자식 테이블에 기본 값
	- DEPENENT : 부모 테이블에 PK가 존재할 때만 자식 테이블에 값이 입력 가능
	- NO ACTION : 제약 조건을 위배한 경우 동작 실패 => 아무 액션 취하지 않음


    (3) UNIQUE KEY : 모든 값이 고유해야 하는 조건 지정, NULL값 허용
    (4) NOT NULL : NULL 값 허용 X
    (5) CHECK : 가질 수 있는 값을 특정 범위로 제한 ex) CHECK(GENDER IN ('M', 'F'))

 2) ALTER : 스키마(컬럼) 변경
    (1) 컬럼 추가 : ALTER TABLE 테이블명 ADD 컬럼명1 데이터타입1; -> 기존 컬럼 끝에 추가
    (2) 컬럼 삭제 : ALTER TABLE 테이블명 DROP COLUMN 컬럼명1; -> 테이블에서 컬럼명1 삭제

    (3) 컬럼 변경 : ALTER TABLE 테이블명 MODIFY(컬럼명1 데이터타입1 제약조건); 
	-> 컬럼에 저장된 데이터가 하나도 없는 경우에만 데이터 타입 변경 가능
	-> 데이터 크기를 줄이는 경우 이미 저장되어 있는 값들을 줄어든 데이터 크기에 모두 담을 수 있을 경우에만 줄일 수 있음
	-> DEFAULT를 지정할 경우, 변경 이후에 추가되는 레코드에 대해서만 기본값이 적용
	-> NOT NULL로 변경하려면 현재 컬럼의 값 중에 NULL이 없어야 가능

	-> SQL 서버 컬럼 변경 : ALTER TABLE 테이블명 ALTER COLUMN 컬럼명1 데이터타입1 => 여러 컬럼을 동시 수정 X

    (4) 컬럼 이름 변경 : ALTER TABLE 테이블명 RENAME COLUMN 컬럼명1 TO 컬럼명2 -> 컬럼명1을 컬럼명2로 변경
    (5) 제약조건 추가 : ALTER TABLE 테이블명 ADD CONSTRAINT 이름1 제약조건(컬럼명) -> 컬럼명에 대해 이름1이란 이름의 제약조건을 추가
    (6) 테이블 삭제 : DROP TABLE 테이블1 [CASCADE CONSTRAINTS]; 
	-> 테이블을 스키마까지 완전하게 삭제
	-> 다른 테이블에서 테이블1을 참조하고 있는 경우 CASCADE CONSTRAINTS를 명시하지 않으면 삭제 불가능
	-> SQL 서버에서는 CASCADE 옵션 사용X => 테이블 삭제 전에 별도로 제약조건 삭제
    (7) 테이블 초기화 : TRUNCATE TABLE 테이블명; -> 테이블의 데이터만 삭제하고 스키마(컬럼명 / 열)은 그대로 둠. DELETE와 달리 롤백이 불가능, 저장공간이 릴리즈 됨

4. DCL : 데이터 제어어
 - 사용자를 생성하고 권한을 부여하는 명령어
 
 1) USER 관련 명령어 : 생성, 변경, 삭제
  (1) USER 생성 : CREATE USER 사용자1 IDENTIFIED BY 패스워드1; -> 사용자 1이란 이름의 USER 생성하고 패스워드1로 비밀번호 지정
  (2) USER 변경 : ALTER USER 사용자1 IDENTIFIED BY 패스워드2; -> 이미 생성되어 있는 사용자1의 비밀번호를 패스워드2로 변경
  (3) USER 삭제 : DROP USER 사용자1; -> 사용자1 삭제

 2) 권한 관련 명령어 : 생성한 계정에 대해 특정 디비에 대한 읽기, 쓰기 등의 접근 권한 부여
  (1) GRANT : GRANT 권한1 [ON 테이블1] TO 사용자1 [WITH GRANT OPTION];
	-> 사용자1에게 권한1을 부여
  	-> 특정 테이블에 대해서만 권한을 부여하려면 ON을 추가
	-> WITH GRANT OPTION을 추가하면 권한을 받는 사용자가 다른 사용자에게 자신이 부여받은 권한을 줄 수 있음
	-> 테이블 생성할 수 있는 권한 부여 : GRANT CREATE TABLE TO 사용자1;

 * 권한 종류
	-> CREATE USER : 사용자 생성
	-> ALTER USER : 생성된 사용자의 정보 변경
	-> DROP USER : 사용자 삭제
	-> CREATE SESSION : 디비 접속 권한
	-> ALTER SESSION : 디비 접속 상태에서 환경 값 변경할 수 있는 권한
	-> CREATE TABLE : 자신 소유의 테이블 생성
	-> CREATE ANT TABLE : 임의의 스키마 소유 테이블 생성

  (2) REVOKE : REVOKE 권한1 [ON 테이블1] FROM 사용자1 [RESTRICT / CASCADE];
	-> 사용자1에게 부여한 권한1을 회수
	-> 특정 테이블에 대해서만 권한 회수는 ON 옵션 쓰기
	-> RESTRICT는 해당 권한을 회수할 때 다른 권한도 같이 회수되어야 하는 경우, 즉 다른 권한에 의존적인 경우에 본 권한 회수 명령 수행 X
	-> CASCADE는 해당 권한을 회수할 때 의존적인 다른 권한까지 함깨 회수
	-> 테이블 생성 권한 회수 : REVOKE CREATE FROM  사용자1;

 3) ROLE 관련 명령어 : 패키지로 묶어서 각 계정에 권한을 부여하는 방식
	(1) CREATE ROLE ROLE_MGR; => ROLE_MGR이라는 이름의 ROLE을 생성
	(2) GRANT CREATE SESSION, CREATE TABLE, CREATE USER TO ROLE_MGR; => 생성된 ROLE_MGR에 세션 연결, 테이블 생성, 사용자 생성 권한 부여
	(3) GRANT ROLE_MGR TO USER1; => ROLE_MGR 역할을 USER1이라는 사용자에게 부여
	(4) 이제 USER1이란 계정은 세션 연결, 테이블 생성, 사용자 생성권한을 가지는 관리자 역할의 계정이 됨
